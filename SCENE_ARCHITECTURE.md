# SCENE ARCHITECTURE — ONE VIEWPORT CINEMATIC JOURNEY

## 1. Core Architecture Rule

This portfolio must not feel like a normal vertical website.

It must not feel like stacked slides.

It must not feel like scrolling down through separate Figma frames.

The final experience must feel like one cinematic viewport where the scene transforms inside the same screen.

The user may scroll, but visually the page should feel like:

* one fixed stage
* one continuous world
* one cinematic timeline
* one viewport transformation
* one connected scene journey

Scroll should control animation progress.

Scroll should not feel like page movement.

The correct experience:

User scrolls → the same viewport stays visually fixed → objects transform → the scene changes inside the screen.

The wrong experience:

User scrolls → the page moves down → another section appears below.

---

## 2. One Viewport Rule

All main portfolio scenes must be designed as part of one connected viewport journey.

The website can internally use multiple components, but the visual experience should feel like one continuous cinematic stage.

Each scene should not appear as a separate stacked section.

Each scene should feel like a transformation state of the same digital environment.

Scene 1 should transform into Scene 2.

Scene 2 should transform into Scene 3.

Scene 3 should transform into Scene 4.

And so on.

Do not create normal vertical page rhythm.

Do not create a layout where each slide is simply below the previous slide.

---

## 3. Scroll Is Timeline Controller

Scroll is only the controller for the animation timeline.

Scroll should control:

* object position
* object scale
* object depth
* title transformation
* scene lighting
* background morphing
* panel transformation
* camera movement
* object docking
* environment reconstruction
* next-scene reveal

Scroll should not primarily control:

* page moving downward
* section entering from below
* ordinary fade-in reveal
* simple slide-up content
* normal landing-page transitions

The user should feel:

“I am controlling a cinematic animation.”

Not:

“I am scrolling down a website.”

---

## 4. Pinned Stage Requirement

Use a pinned or sticky stage architecture for main scene transitions.

Recommended structure:

* an outer scroll container provides scroll distance
* an inner stage remains sticky / pinned
* the inner stage fills the viewport
* scroll progress controls animation inside the stage

Example concept:

outer scene journey container
→ long scroll height
→ sticky viewport stage
→ animated scene objects inside
→ progress 0 to 1 controls all transformations

Do not let the visual viewport simply move down from one slide to another.

---

## 5. Scene Morph Requirement

Transitions between scenes must be real transformations.

A transition is not complete if it only uses:

* opacity
* fade
* slide
* scale
* tilt
* parallax
* random data lines
* scan beams
* floating labels

A real transition should include object role change.

Examples:

* identity panels collapse into terminal sockets
* lanyard transforms from hero photo card into access key
* monitor wall opens into command center
* command center terminals rearrange into knowledge library shelves
* terminal blocks become book/module cards
* book pages extrude into city streets
* buildings compress into project cartridges
* project cartridges transform into achievement metrics
* trophy fragments become knowledge nodes
* knowledge nodes collapse into future data core

The important rule:

Objects from the old scene must become objects in the next scene.

Do not just hide one scene and show another.

## 13. Forbidden Global Pattern

Never build the portfolio like this:

Slide 1 section
↓
Slide 2 section
↓
Slide 3 section
↓
Slide 4 section

This is forbidden.

Never make the user feel like they are reading a normal web page.

Never make the transition feel like scrolling down a Figma mockup.

Never rely on “the next section appears below” as the main experience.

---

## 14. Required Global Pattern

Build the experience like this:

One cinematic viewport stage
→ Scene 1 normal state
→ Scene 1 transforms into Scene 2
→ Scene 2 stabilizes
→ Scene 2 transforms into Scene 3
→ Scene 3 stabilizes
→ Scene 3 transforms into Scene 4
→ continue until Scene 8

The user should feel like they are moving through one continuous animated system.

---

## 15. Transition Quality Standard

A transition is acceptable only if:

* it keeps the viewport visually fixed
* it uses scroll as animation control
* old scene objects transform into new scene objects
* the new scene forms inside the same screen
* title/content changes feel animated, not replaced
* the user does not feel like the page simply moved down
* the final state feels like a different scene
* the transition feels intentional and cinematic

A transition is not acceptable if:

* it only tilts panels
* it only fades elements
* it only slides elements
* it only zooms
* it only adds scan lines
* it only adds decorative particles
* it only shows labels
* it moves to a new section below
* it feels like Figma scroll

---

## 16. Visual Style During Transitions

Transitions must stay premium.

Use:

* dark atmosphere
* controlled lighting
* restrained teal/cyan palette
* subtle emerald status accents
* tiny gold highlights only when needed
* mechanical motion
* morphing objects
* depth movement
* environment reconstruction
* cinematic timing

Avoid:

* colorful neon
* noisy beams
* random lines
* cheap sci-fi UI
* too many labels
* childish animation
* crowded overlays
* excessive glow

Premium means restrained, precise, and intentional.

---

## 17. Implementation Reminder

Before building any new scene, first confirm the transition architecture.

Do not build Scene 2 as a normal section below Scene 1.

Do not build Scene 3 as a normal section below Scene 2.

Create the global journey architecture first.

The main technical goal is:

A scroll-controlled pinned cinematic stage.

The first technical milestone should be:

Scene 1 transforms into a command-center shell inside the same viewport.

Only after this transition feels correct should full Scene 2 content be added.

---

## 18. Acceptance Checklist

Before accepting any transition, check:

1. Does the viewport stay visually fixed?
2. Does the user feel like they are still inside one screen?
3. Does scroll control animation rather than page movement?
4. Do old objects transform into new objects?
5. Does the scene actually change, not just tilt?
6. Does the title transform into the next title?
7. Do panels morph into next-scene modules?
8. Does the background reconstruct into the next environment?
9. Does the final state look like a new scene?
10. Does it avoid normal vertical section scrolling?
11. Does it avoid noisy scan/data-line gimmicks?
12. Does it feel premium and cinematic?

If the answer is no, the transition is not accepted.
