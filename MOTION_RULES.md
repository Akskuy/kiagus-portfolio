# MOTION RULES — NON-BASIC ANIMATION CONTRACT

## 1. Absolute Motion Rule

No important element is allowed to feel dead.

Every major object must feel alive, interactive, and intentionally animated.

This applies to:

* hero identity card
* lanyard card
* avatar/photo frame
* panels
* title-only panels
* buttons
* cards
* terminals
* monitors
* data lines
* background screens
* section titles
* progress rails
* navigation nodes
* city buildings
* project cartridges
* scanner machine
* achievement cards
* trophy objects
* knowledge nodes
* data core
* final CTA station
* placeholders
* separators
* status chips
* icon tiles
* micro labels

Every important object must have at least one visible life effect.

Acceptable life effects:

* breathing glow
* pixel pulse
* scanline movement
* terminal cursor blink
* moving data packets
* animated border trace
* ambient particle drift
* soft parallax drift
* hover expansion
* click activation
* status LED blinking
* number ticking
* node pulse
* cable data flow
* moving light reflection
* UI flicker
* layered depth movement
* scanner beam
* object transformation
* hologram shimmer
* circuit-line activation

Motion must be visible enough to make the interface feel alive.

However, motion must stay controlled, premium, readable, and professional.

Do not make motion chaotic.

---

## 2. Main Animation Philosophy

The animation language must not feel like a normal website.

Do not rely on:

* fade in
* slide up
* slide left
* slide right
* scale in
* basic hover shadow
* basic dropdown
* plain accordion
* opacity change only
* normal scroll reveal
* static section with one hover effect
* hard cut between sections

These may be used only as tiny supporting details.

They cannot become the main animation language.

The main animation language must use:

* object activation
* system boot
* scanner reveal
* terminal execute
* mechanical hatch unlock
* hologram projection
* data compile
* pixel assembly
* x-ray scanner reveal
* circuit activation
* lanyard access scan
* cartridge insertion
* node connection drawing
* object morph
* transition gates
* scroll-controlled scene progression
* ambient interface life

The user should feel:

“This was designed like a real interactive system, not generated as a basic website.”

---

## 3. No Plain Vertical Scroll Rule

The website must not feel like stacked vertical sections.

Scroll is not just page movement.

Scroll should act as a scene controller.

Do not create a website where the user simply scrolls down through separate sections.

Avoid this feeling:

* section appears
* content fades in
* user scrolls
* next section appears
* repeat

Use this feeling instead:

* scene boots
* objects activate
* panels unlock
* camera/environment responds
* transition gate prepares
* old scene transforms into the next scene

Scroll can control:

* camera depth
* scene activation
* object transformation
* UI state progression
* light intensity
* parallax layers
* transition gate movement
* data flow speed
* scanner progress
* node connection progress
* scene reconstruction

Each scene should feel like a pinned or controlled interactive game screen where appropriate.

The final user experience should feel like moving through a connected data-science game journey.

---

## 4. Scene Motion Phase Structure

Every scene should have 4 internal phases.

## Phase A — Scene Entry

The scene starts from the previous scene’s transition object.

Examples:

* identity card scan data becomes command center
* terminal UI fragments become knowledge library blocks
* book page becomes experience city
* building compresses into project cartridge
* project scanner output becomes achievement analytics
* trophy reflections become knowledge nodes
* knowledge nodes collapse into future data core

The scene should not start as a hard cut.

## Phase B — Scene Boot

The scene activates like a system.

Examples:

* background lights turn on
* monitor pixels flicker
* terminal cursor starts blinking
* data cables begin flowing
* status LEDs activate
* panels receive border traces
* scanlines sweep across UI
* small system labels appear
* background depth becomes visible

Scene boot must feel like a live interface starting up.

## Phase C — Main Interactive State

The scene becomes usable.

Required:

* hover states
* tap/click states
* active state
* inactive state
* title-only interactive panels
* creative panel reveal
* readable description content
* ambient idle motion
* non-static UI details

The scene must not feel like a static screenshot.

## Phase D — Transition Gate

A specific object prepares the transition into the next scene.

Examples:

* lanyard card sends data to monitor wall
* terminal deconstructs into data blocks
* book opens into city map
* building compresses into cartridge
* scanner output becomes achievement dashboard
* trophy shatters into nodes
* nodes collapse into data core

The transition gate must be object-based.

---

## 5. Interactive Panel Reveal Rule

This is one of the most important rules.

Many portfolio details should start as title-only panels.

The user clicks a title-only panel to reveal the description.

This reveal must not be basic.

Do not open panels with:

* plain accordion
* simple dropdown
* simple fade
* simple slide down
* simple scale
* instant open
* ordinary card expansion
* plain flip only

When a title-only panel is clicked, the reveal must feel like a system interaction.

Every important panel reveal must have 4 stages:

## Stage 1 — Trigger

The click activates something visible.

Possible trigger effects:

* LED turns on
* status chip changes
* small icon pulses
* scanner line appears
* command cursor activates
* lock indicator unlocks
* node pulses
* cartridge slot lights up
* page edge glows
* building window turns on

## Stage 2 — Transformation

The panel physically or visually changes.

Possible transformation effects:

* frame expands
* corner brackets unlock
* panel opens like a hatch
* border traces around the card
* page unfolds
* data chip inserts
* hologram projection rises
* terminal output area expands
* scanner beam crosses the panel
* pixel blocks assemble
* circuit lines connect

## Stage 3 — Content Reveal

The description appears through a non-basic reveal.

Possible content reveal effects:

* terminal output appears line by line
* text appears after scanline passes
* hologram text stabilizes
* encrypted/glitched text resolves
* data chips compile into paragraphs
* x-ray layer reveals hidden detail
* pixel blocks assemble into description
* page content writes itself in structured rows

## Stage 4 — Active Idle

After opening, the panel remains alive.

Possible active idle effects:

* subtle glow pulse
* status LED breathing
* border trace loop
* cursor blink
* scanline movement
* data line movement
* endpoint glow
* active corner shimmer
* tiny particle drift
* active selected state

Opened panels must feel active, not static.

3. scanline passes over title

2. page unlocks
3. page opens or unfolds

## 8. Transition Creativity Rule

Scene transitions must be creative.

Do not overuse zoom.

Zoom may be used as a supporting camera movement, but it must not be the main idea of every transition.

The main transition should come from object transformation.

The transition should make the user feel:

“How was this made?”

Do not use:

* hard cut
* plain fade
* fade to black
* basic slide transition
* basic zoom transition repeated everywhere
* normal section reveal
* jump scroll

Every transition must include:

* object continuity
* transformation
* motion direction
* lighting change
* depth change
* connection between old scene and new scene
* visible transition gate
* controlled scroll or interaction progress

Important micro-elements include:

* border lines
* separators
* status dots
* chips
* icon tiles
* progress rails
* panel corners
* title frames
* miniature badges
* card corners
* small arrows
* data dividers
* CTA edges
* placeholder borders
* thumbnail frames
* terminal row indicators
* node endpoints
* scrollbar/scene rail indicators

Acceptable micro motion:

* faint glow pulse
* moving data packet
* border trace
* scanline pass
* LED blink
* hover shimmer
* active node pulse
* tiny parallax offset
* endpoint glow
* small flicker
* segmented line animation
* corner light pulse
* circuit signal travel

Do not animate every micro-element aggressively.

But avoid dead, flat, generic UI.

The interface should feel alive at macro level and micro level.

## 11. Game Interface Motion Rule

This portfolio should feel like a polished game interface.

Panels should not only appear.

Panels should activate.

Borders should not only exist.

Borders should trace, pulse, or respond.

Lines should not only divide.

Lines should carry data or glow at endpoints.

Buttons should not only hover.

Buttons should feel like game controls.

Cards should not only sit.

Cards should breathe, flicker, or hold energy.

Nodes should not only connect.

Nodes should transfer signals.

Every scene should feel like a live interface system.

## 18. Scroll Animation Foundation

Each scene should expose or use a scroll progress value from 0 to 1 when useful.

Scroll progress can control:

* camera scale
* camera translate
* layer depth
* light intensity
* object opacity
* object transform
* panel activation
* data flow speed
* blur intensity
* scanline intensity
* transition state
* scene boot progress
* transition gate progress

Use GSAP ScrollTrigger for scroll-controlled sequences.

Use Framer Motion for:

* hover states
* tap states
* click states
* component state changes
* panel reveal
* active/inactive UI states

Use CSS animations for:

* scanlines
* blinking cursor
* subtle glows
* looping data packets
* pixel particles
* LED blinking
* background flicker

Do not force heavy animation when simple CSS motion is enough.

---

## 19. Idle Motion Rule

Even when the user does nothing, each scene must still feel alive.

Every scene must include ambient idle motion, such as:

* slowly moving particles
* glowing data paths
* flickering terminal pixels
* floating UI noise
* blinking cursors
* node pulses
* breathing panels
* tiny data stream movement
* animated background grid
* subtle parallax drift
* monitor-wall flicker
* status LED blink

Idle motion should be subtle and premium.

Do not make idle motion annoying.

---

## 20. Performance Rule

The motion system must be impressive but still usable.

Avoid creating too many heavy animations at once.

Avoid unnecessary expensive canvas effects if CSS/SVG can do the job.

Prioritize:

1. smooth interaction
2. readable text
3. responsive layout
4. premium motion feel
5. controlled visual density

Do not sacrifice usability for animation.

---

## 21. Responsive Motion Rule

## Desktop

Use richer motion:

* pinned scene sections where useful
* scroll-controlled transitions
* multi-layer parallax
* hover states
* richer UI density
* deeper scene composition

## Tablet

Reduce density:

* preserve main scene identity
* use tap instead of hover
* reduce heavy parallax
* keep important animations
* keep text readable

## Mobile

Simplify motion:

* avoid horizontal overflow
* reduce heavy particles
* avoid complex pinned sequences if unstable
* use tap interactions
* keep core panel reveal behavior
* keep important life effects
* keep layout readable

Mobile must not become a broken desktop layout.

---

## 22. Reduced Motion Rule

Support prefers-reduced-motion.

If reduced motion is enabled:

* disable heavy camera movement
* reduce parallax
* reduce continuous particles
* reduce aggressive flicker
* simplify transition gates
* keep website readable and usable
* preserve lightweight UI states
* keep subtle glow, cursor blink, and basic active states

Do not remove all interactivity.

Reduced motion should still feel polished.

---

## 23. Implementation Safety Rule

Do not create animation that causes hydration errors.

Do not use browser-only APIs during server render without proper guards.

For browser-only animation libraries:

* use client components
* guard window/document access
* initialize GSAP/ScrollTrigger safely
* clean up animation instances
* avoid random values during SSR unless generated safely on client
* avoid layout shift caused by late animation setup

Motion should not break build, lint, or typecheck.

---

## 24. Final Motion Checklist

Before considering any scene complete, check:

1. Does the scene avoid basic fade/slide as the main animation?
2. Does the scene feel alive before interaction?
3. Are important objects animated?
4. Are micro-elements subtly alive?
5. Do panels feel activated rather than dead?
6. Do title-only panels reveal content creatively?
7. Does panel reveal have trigger, transformation, content reveal, and active idle?
8. Do buttons feel like game controls?
9. Do cards feel like custom game UI modules?
10. Does the hero object feel valuable and interactive?
11. Does the background have scene-specific life?
12. Does hover/tap create meaningful response?
13. Is the transition gate object-based?
14. Is zoom not overused?
15. Does scroll feel like scene control, not normal page scrolling?
16. Is motion controlled and professional?
17. Is the content still readable?
18. Does layout fit at 100% browser zoom?
19. Does it avoid fake technical claims?
20. Does it feel difficult to build?

Final motion goal:

The portfolio should feel like a premium interactive data-science game interface, not a normal animated landing page.
